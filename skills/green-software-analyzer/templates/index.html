Análise de Design Patterns Green Software

Avaliação do impacto potencial da implementação de 4 padrões específicos de design para eficiência energética.

1. Green by Default

Configurações padrão otimizadas para o menor consumo de recursos possível.

Status Atual: 15.498 queries usam limit=-1 e 68.713 não definem paginação explícita.

Potencial de Redução de Payload45%

Ao forçar limit=50 como padrão e remover limit=-1, eliminamos o desperdício de processar dados que o cliente muitas vezes não exibe.

2. Just Latest Updates (Delta)

Entrega apenas a diferença (delta) entre a última requisição e a atual.

Candidatos: 71,6% das requests (149.135) são repetições de recursos em menos de 5 min.

Economia de Banda Estimada70%

Com intervalo médio de 22s entre polls, o uso de deltas evitaria o reenvio constante de dados estáticos (ex: nomes de hotéis, coordenadas).

3. Wish List (Sparse Fieldsets)

Permite ao cliente selecionar exatamente quais campos deseja receber.

Uso Atual: 50,7% já usam fields=. 102.663 ainda recebem o objeto completo.

Redução de Payload por Request60%

Campos como Id e Title são os mais pedidos. Tornar a seleção obrigatória reduz o custo de serialização JSON.

4. Wish Template

Beneficiários: 20 endpoints principais concentram 90% da carga de serialização.

SCI — Software Carbon Intensity (Green Software Foundation)

📐 Fórmula SCI (GSF Specification v1)

SCI = ((E_net + E_compute) × I + M) / R

E_net = bytes transferidos × 0,001 kWh/GB  ·  E_compute = 100 W × latência  ·  I = 233 gCO₂eq/kWh (Itália 2023, Ember Climate)  ·  M = 0,000013 gCO₂eq/req (hardware amortizado)  ·  R = 1 request (unidade funcional)

Recomendações Green Software

Estratégias priorizadas para reduzir o impacto ambiental da plataforma, baseadas nos padrões identificados nos logs.

1

Implementar Cache HTTP com TTL Adequado

Adicionar headers Cache-Control: max-age=60 e suporte a ETag / Last-Modified nos endpoints de alta repetição. Endpoints como /v1/STA/Accommodation (ratio 1664x) e /v1/STA/ODHActivityPoi (ratio 1365x) devem ser priorizados.

Redução: até 61,3% Impacto: Crítico Esforço: Médio
2

Eliminar ou Controlar Queries com limit=-1

As 15.498 queries com limit=-1 (7,4% do total) retornam todos os registros sem paginação, concentradas nos hosts de mobilidade. Implementar paginação obrigatória ou limites máximos (ex: limit=1000) reduz drasticamente o payload de resposta.

Redução de banda: ~40% Impacto: Alto Esforço: Baixo
3

Implementar Debounce e Rate Limiting nos Clientes

29,6% das requests são duplicatas no mesmo segundo. Implementar debounce de 1-5 segundos nos web components e rate limiting por IP/origem eliminaria o polling agressivo sem impactar a experiência do usuário.

Redução: 29-43% Impacto: Alto Esforço: Médio
4

Consolidar Hosts Legados

Existem 4 hosts legados (*.bz.it e tourism.opendatahub.com) que somam 34% do tráfego. Migrar para os hosts atuais (*.api.opendatahub.com) e desativar os legados reduz overhead de DNS, certificados TLS e manutenção de infraestrutura.

Redução infra: ~34% Impacto: Médio Esforço: Médio
5

Implementar Seleção de Campos (Sparse Fieldsets)

Muitas requests usam o parâmetro fields= para selecionar apenas alguns campos, mas a maioria não o usa. Incentivar o uso de fields= e implementar projeção no servidor reduz o tamanho das respostas e o processamento de serialização.

Redução payload: 30-70% Impacto: Médio Esforço: Baixo
6

Cache por Idioma com Vary Header

60,2% das requests especificam idioma (DE, IT, EN). Implementar cache com Vary: Accept-Language ou incluir idioma na chave de cache permite reutilizar respostas por idioma, reduzindo processamento de tradução/filtragem no servidor.

Redução CPU: ~25% Impacto: Moderado Esforço: Baixo
