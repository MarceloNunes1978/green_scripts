Green Software Design Patterns Analysis

Assessment of the potential impact of implementing 4 specific design patterns for energy efficiency.

1. Green by Default

Default configurations optimized for the lowest possible resource consumption.

Current Status: 15,498 queries use limit=-1 and 68,713 do not define explicit pagination.

Payload Reduction Potential45%

By enforcing limit=50 as default and removing limit=-1, we eliminate the waste of processing data that the client often does not display.

2. Just Latest Updates (Delta)

Delivers only the difference (delta) between the last request and the current one.

Candidates: 71.6% of requests (149,135) are resource repetitions within less than 5 min.

Estimated Bandwidth Savings70%

With an average interval of 22s between polls, using deltas would avoid the constant resending of static data (e.g., hotel names, coordinates).

3. Wish List (Sparse Fieldsets)

Allows the client to select exactly which fields they want to receive.

Current Usage: 50.7% already use fields=. 102,663 still receive the full object.

Payload Reduction per Request60%

Fields like Id and Title are the most requested. Making selection mandatory reduces the JSON serialization cost.

4. Wish Template

Pre-defined templates for common use cases (e.g., 'summary', 'map').

Beneficiaries: Top 20 endpoints account for 90% of the serialization load.

Server CPU Reduction25%

Templates avoid repetitive parsing of field strings and enable cache optimizations at the database projection level.

Green Software Recommendations

Prioritized strategies to reduce the environmental impact of the platform, based on patterns identified in the logs.

1

Implement HTTP Cache with Adequate TTL

Add Cache-Control: max-age=60 headers and support for ETag / Last-Modified on high-repetition endpoints. Endpoints like /v1/STA/Accommodation (ratio 1664x) and /v1/STA/ODHActivityPoi (ratio 1365x) should be prioritized.

Reduction: up to 61.3% Impact: Critical Effort: Medium
2

Eliminate or Control Queries with limit=-1

The 15,498 queries with limit=-1 (7.4% of total) return all records without pagination, concentrated on mobility hosts. Implementing mandatory pagination or maximum limits (e.g., limit=1000) drastically reduces the response payload.

Bandwidth reduction: ~40% Impact: High Effort: Low
3

Implement Debounce and Rate Limiting on Clients

29.6% of requests are duplicates within the same second. Implementing 1-5 second debounce in web components and rate limiting per IP/origin would eliminate aggressive polling without impacting user experience.

Reduction: 29-43% Impacto: Alto Esforço: Médio
4

Consolidate Legacy Hosts

There are 4 legacy hosts (*.bz.it and tourism.opendatahub.com) that account for 34% of traffic. Migrating to current hosts (*.api.opendatahub.com) and decommissioning legacy ones reduces DNS overhead, TLS certificates, and infrastructure maintenance.

Infra reduction: ~34% Impact: Medium Esforço: Médio
5

Implement Field Selection (Sparse Fieldsets)

Many requests use the fields= parameter to select only a few fields, but most do not. Encouraging the use of fields= and implementing server-side projection reduces response size and serialization processing.

Payload reduction: 30-70% Impacto: Médio Esforço: Baixo
6

Language Cache with Vary Header

60.2% of requests specify a language (DE, IT, EN). Implementing cache with Vary: Accept-Language or including language in the cache key allows reusing responses per language, reducing translation/filtering processing on the server.

CPU reduction: ~25% Impact: Moderate Esforço: Baixo
